## 1. Overview

Bigood is a SaaS platform for barbershop management, built as a **frontend-only high-fidelity prototype**. It comprises three main areas:

1. **Institutional Landing Page** — lead generation for consultative B2B sales
2. **Barber/Admin Dashboard** — full management panel
3. **Client Portal** — mobile-first customer app for scheduling and subscriptions

No real backend exists. All data is mocked via localStorage and an in-memory mock database.

## 8. Authentication Approach

### Dual Authentication System

**System A — Barber/Admin Auth (Cookie-based HMAC):**

- File: lib/auth.ts
- Cookie name: mydashbarber.session
- Session max age: 8 hours (AUTH_SESSION_MAX_AGE = 60 * 60 * 8)
- Token format: ase64UrlEncode(payload).HMAC-SHA256
- HMAC key: process.env.AUTH_SECRET (default: "troque-por-um-segredo-longo-em-producao")
- Payload: { email, name, companyName, hasActivePlan, planKey, sub, exp }
- Security: safeEqual() for timing-safe signature comparison
- Environment variables: ADMIN_EMAIL, ADMIN_PASSWORD (single admin only)

API Routes:
- POST /api/auth/login — validates credentials, sets cookie
- POST /api/auth/logout — clears cookie
- GET /api/auth/session — reads cookie, returns user or null
- POST /api/auth/signup — creates session cookie (no persistence)
- POST /api/billing/activate — upgrades session to hasActivePlan: true

**Route Protection** (proxy.ts middleware):
- Dashboard routes (/agenda, /assinaturas, /caixa, /clientes, /dashboard, /empresa, /financeiro, /planos, /profissionais, /servicos) require authenticated session with hasActivePlan
- Subscription routes (/checkout, /escolher-plano) require auth but no plan
- Account route (/conta) requires auth
- Login redirects with ?next= parameter
- Matcher: all routes except static assets

**System B — Client/Customer Auth (localStorage-based):**

- File: services/client-auth.ts
- Storage key pattern: igood.clientPortal.{slug}.auth
- Users stored at: igood.clientPortal.{slug}.auth.users
- Session stored at: igood.clientPortal.{slug}.auth
- No server-side validation — entirely client-side mock
- Token generation: 	ok_{timestamp}_{random}
- Password stored in plain text in localStorage
- Simulated network delay (wait new Promise(r => setTimeout(r, 600)))
