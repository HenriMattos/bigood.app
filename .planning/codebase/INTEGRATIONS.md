## 1. External Services and APIs

### Current Status: None

Bigood is a **frontend-only prototype** with no real external service integrations. All data persistence, authentication, and business logic run entirely client-side or via mock API routes within the Next.js application itself.

**No integrations exist for:**
- Payment gateways (Stripe, Mercado Pago, etc.)
- SMS/email providers (Twilio, SendGrid, etc.)
- Authentication providers (Auth0, Clerk, NextAuth, etc.)
- Analytics (GA4, Mixpanel, etc.)
- Error tracking (Sentry, etc.)
- Database (PostgreSQL, Supabase, etc.)
- File storage (S3, Cloudinary, etc.)
- Maps/geolocation (Google Maps, etc.)
- Social media APIs (Instagram, WhatsApp Business API)
- Calendar sync (Google Calendar, Outlook)

## 2. localStorage Usage Patterns

localStorage is the primary persistence layer across the entire application. There are **three distinct usage patterns**:

### Pattern A: Client Portal Auth Storage

**File**: services/client-auth.ts

**Storage keys (dynamic per barbershop slug):**
- igood.clientPortal.{slug}.auth — current session { user, token }
- igood.clientPortal.{slug}.auth.users — all registered users { ...user, password }[]

**Usage flow:**
1. egisterClient() — writes new user to users array, writes session
2. loginClient() — reads users array, validates credentials, writes session
3. logoutClient() — removes session
4. getClientSession() — reads session, returns user + token

**Security concerns:**
- Passwords stored in plain text (no hashing)
- Token is a mock (	ok_{timestamp}_{random})
- Entirely client-side — no server validation
- Any JavaScript running on the page can read all data

### Pattern B: Client Portal State Storage

**File**: components/client-portal/client-portal-data.ts

**Storage key:** igood.clientPortal.{slug}.state

**Data stored:** Full client portal state including:
- profile (name, phone, email, birthDate, cpf, gender, contactPreferences)
- ppointments[] (service, professional, date, time, price)
- subscription (plan, status, dates, payment method)

**Usage flow:**
1. eadClientPortalState() — reads on page load, merges with defaults
2. saveClientPortalState() — auto-saves via useEffect on every state change
3. Default state: pre-populated with demo data (Gabriel Silva, demo appointment)

### Pattern C: Company Configuration Storage

**Files**: components/company/company-assets.ts, components/company/commercial-storage.ts

**Storage keys (static prefix pattern):**
- mydashbarber.v1.logoUrl
- mydashbarber.v1.iconUrl
- mydashbarber.v1.carouselImage1 through 3
- mydashbarber.v1.clientPortal.bannerUrl
- mydashbarber.v1.clientPortal.description
- mydashbarber.v1.clientPortal.enabled
- mydashbarber.v1.clientPortal.openingHours
- mydashbarber.v1.clientPortal.slogan
- mydashbarber.v1.clientPortal.slug
- mydashbarber.v1.commercial.plans
- mydashbarber.v1.commercial.subscriptions

**Versioned keys:** The 1 in the key prefix indicates an intentional versioning strategy for future migration or clearance.

**Usage flow:**
1. Admin dashboard reads/writes these values
2. Client portal reads them to configure its display
3. eadStringStorage() / writeStorage() helper functions with SSR guards
4. Fallback to database.ts defaults when localStorage is empty

### Pattern D: Auth Sync Events (Cross-tab Communication)

The application uses the **Web Storage API's storage event** for cross-tab synchronization, plus custom events for in-page communication:

`
window.addEventListener("storage", syncAuth);          // use-auth.tsx
window.addEventListener("bigood_auth_sync", syncAuth); // use-auth.tsx
window.addEventListener("storage", handleStorage);      // client-portal-shell.tsx
window.addEventListener(COMPANY_PORTAL_SYNC_EVENT, refreshPortalData);  // client-portal-shell.tsx
window.dispatchEvent(new Event("bigood_auth_sync"));   // use-auth.tsx, admin-shell.tsx
`

**Note**: The native storage event only fires in other tabs (not the tab that made the change), so the custom igood_auth_sync event is dispatched in the same tab after auth state changes.

## 3. Custom Events

### igood_auth_sync

**Purpose**: Synchronize authentication state across all components in the same tab + cross-tab.

**Dispatched from:**
- hooks/use-auth.tsx:
  - setAuthenticatedUser() — after login
  - logout() — after logout
  - useEffect mount — sets up listener
- components/admin/admin-shell.tsx:
  - logout() function — after POST /api/auth/logout

**Listened by:**
- hooks/use-auth.tsx — triggers efresh() to re-fetch session from API

**Event type**: Standard Event (no custom data)

### COMPANY_PORTAL_SYNC_EVENT

**Constant**: "mydashbarber.clientPortal.sync" (defined in components/company/company-assets.ts)

**Purpose**: Notify the client portal to refresh its data when company settings change in the admin dashboard.

**Dispatched from:** (not dispatched by any component yet — prepared for future admin-to-portal sync)

**Listened by:**
- components/client-portal/client-portal-shell.tsx — triggers efreshPortalData() which re-reads from localStorage

**Event type**: Standard Event (no custom data)

### storage (Native Web API)

Used for cross-tab synchronization in:
- use-auth.tsx — re-fetches auth session
- client-portal-shell.tsx — refreshes portal data when localStorage key includes clientPortal
- dmin-shell.tsx — syncs company data changes
