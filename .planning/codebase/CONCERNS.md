## HIGH SEVERITY

### H1. No Real Backend — All Mock Data and localStorage

**What:** The entire application runs on hardcoded mock data and browser localStorage. There is no real database, API server, or persistent backend.

**Evidence:**
- components/admin/database.ts (645 lines) — single file with all mock data for company, services, plans, clients, subscriptions, professionals, agenda events, comandas, payment methods, financial data.
- components/client-portal/client-portal-data.ts (331 lines) — custom data layer that reads from database.ts and localStorage keys like bigood.clientPortal.{slug}.*
- components/company/commercial-storage.ts (65 lines) — localStorage read/write utilities for plans and subscriptions.
- All API routes (app/api/) are thin wrappers that return mock responses with no database layer.

**Risk:** Every feature that appears functional will break when localStorage is cleared, state is stale, or data needs to persist across sessions. No data survives a browser cache clear. No server-side validation exists.

---

### H2. Senhas Armazenadas em Plain Text no localStorage

**What:** Client portal passwords are stored and compared in plain text with zero hashing or salting.

**Evidence:**
- services/client-auth.ts line 87: const storedUser: StoredUser = { ...user, password: input.password }
- Login comparison at line 109: u.password === input.password
- The storage key bigood.clientPortal.{slug}.auth.users stores an array of { ...ClientAuthUser, password: string } in plain text.
- The spec itself acknowledges this: docs/specs/client-auth.spec.md line 99: 'Isso é temporário e não é segurança real.'

**Risk:** Anyone with access to the browser's localStorage inspector can read all client passwords for a given barbershop slug. This is a data privacy and security violation.

---

### H3. Single Hardcoded Admin Credential for Barber Login

**What:** The admin/barber authentication relies on a single hardcoded credential pair, stored in environment variables with weak defaults committed to .env.

**Evidence:**
- lib/auth.ts lines 22-25: getAdminCredentials() returns { email: process.env.ADMIN_EMAIL ?? 'admin@empresa.com', password: process.env.ADMIN_PASSWORD ?? 'admin123' }
- .env file (committed to repository):
  - AUTH_SECRET='troque-por-um-segredo-longo-em-producao'
  - ADMIN_EMAIL='admin@empresa.com'
  - ADMIN_PASSWORD='admin123'

**Risk:** A single hardcoded credential with a weak password means no multi-user support, no role-based access (RBAC), and no audit trail. Any compromise of this single credential grants full system access. The .env file containing the default password is tracked in git.

### H4. Auth is localStorage-Based — Not Real Security

**What:** Client portal authentication stores the entire session (user + token) in plain localStorage rather than using HTTP-only cookies, JWTs, or any server-validated mechanism.

**Evidence:**
- services/client-auth.ts — all session data stored under bigood.clientPortal.{slug}.auth as a JSON blob.
- Token generation uses non-cryptographic random string: generateToken() returns tok__
- Sessions are restored purely from localStorage with no server-side validation (getClientSession).
- The admin auth (lib/auth.ts) uses a better approach (HMAC-signed cookie), but the signup route (api/auth/signup/route.ts) accepts any request without checking credentials against a real user store.

**Risk:** Session tokens can be forged, stolen via XSS, or manipulated. There is no real authentication boundary between client, admin, or company. The signup endpoint provisions access without verification.

### H5. No Tests — Zero Test Infrastructure

**What:** The codebase has no tests whatsoever — no unit tests, integration tests, E2E tests, or test configuration.

**Evidence:**
- package.json has no test-related scripts, dependencies, or configuration.
- No jest.config.*, vitest.config.*, cypress/, playwright/, or __tests__/ directories found.
- No *.spec.*, *.test.*, *test* pattern files found (except docs/specs/client-auth.spec.md which is a documentation spec, not a test).
- devDependencies in package.json: only ESLint, Prettier, Tailwind, TypeScript.

**Risk:** There is zero regression safety. Any refactoring, backend integration, or dependency upgrade carries unknown breakage risk. Spec-driven development cannot be verified without tests. The frontend has complex state logic in components like client-portal-shell.tsx (1190 lines) and agenda-view.tsx (2145 lines) that will be extremely risky to modify without a test suite.

### H6. No Error Boundaries, No Monitoring, No Logging

**What:** The application has no React error boundaries, no crash reporting, no application logging, and no performance monitoring.

**Evidence:**
- No error-boundary* files found anywhere in the codebase.
- No Sentry, Datadog, LogRocket, or similar monitoring packages in package.json.
- No console.* abstraction or structured logging utility.
- No app/error.tsx or app/global-error.tsx files (Next.js conventions for error handling).
- No catch-all error handling in API routes — all routes return NextResponse.json(...) without try/catch wrappers.

**Risk:** Every runtime error silently fails or crashes the UI. There is no observability into production issues. Bugs affecting real users cannot be diagnosed without reproduction steps. The application cannot meet any reasonable SLO or SLA.

### H8. Dashboard and Portal Are Tightly Coupled to Mock Data

**What:** Components directly import from database.ts and client-portal-data.ts instead of going through service abstractions.

**Evidence:**
- components/admin/admin-shell.tsx line 17: import { database } from '@/components/admin/database'
- components/admin/dashboard-view.tsx lines 3-4: imports from database.ts and date-utils.ts
- components/client-portal/client-portal-shell.tsx lines 17-31: imports data functions from client-portal-data.ts

### H9. Portal Has No Real Session/Cookie for Client

**What:** The client portal authentication is entirely client-side, with no server session, no HTTP-only cookie, and no backend validation.

**Evidence:**
- services/client-auth.ts — all operations are pure localStorage read/write.
- hooks/use-client-auth.tsx — state is initialized from localStorage via authService.getClientSession(slug).
- Client session is never sent to a server for validation. The app/api/ has no routes for client auth validation.
- No cookie is set for client portal sessions — only the admin auth (lib/auth.ts) uses HTTP-only cookies.

**Risk:** There is no way to revoke client sessions server-side. A compromised client token cannot be invalidated. There is no companyId/customerId isolation enforced by the server. The client spec explicitly documents this as temporary.

### H10. No types/, services/, schemas/ Directories Until Recently

**What:** These architectural directories were only created very recently and are nearly empty.

**Evidence:**
- types/client-auth.ts — 28 lines (single types file)
- services/client-auth.ts — 134 lines (single service file, still uses localStorage)
- schemas/client-auth.ts — 56 lines (validation functions, no Zod)

- Zod is not a dependency — schemas are plain functions.
- The types/, services/, schemas/ directories do not cover the dashboard, landing, billing, agenda, or any other domain.

**Risk:** The architectural intent is clear (SDD + separation of concerns), but the migration is barely started. The bulk of the codebase still violates the intended architecture.

### H11. No Real API Contracts Implemented

**What:** Only 5 thin API route files exist — none implement real CRUD operations for business entities.

**Evidence:**
| Route | Lines | Purpose | Real? |
|-------|-------|---------|-------|
| api/auth/login/route.ts | 37 | Admin login | Mock — validates against env var |
| api/auth/signup/route.ts | 37 | Admin signup | Mock — accepts anything |

**Risk:** Every entity in the system lacks a defined API contract. Backend development must create these from scratch. The frontend has no integration points to test against.

### H12. Appointment Scheduling Has No Conflict Detection

**What:** The client portal shell creates appointments in-memory without checking for overlapping time slots with existing appointments.

**Evidence:**
- components/client-portal/client-portal-shell.tsx lines 165-183: createAppointment pushes a new appointment to local state with no pre-flight or deconfliction.
- No function calls agendaEvents or database.agendaEvents to check for conflicts.
- The admin scheduling (agenda-view.tsx) also lacks server-side conflict validation — it operates on the same in-memory mock array.

**Risk:** Double-booking is currently possible. When backend integration happens, appointment conflict validation must be implemented server-side. The current mock gives a false sense of working scheduling.

---

### H13. No Multitenancy Isolation (companyId/unitId)

**What:** The mock data assumes a single company ('Bigood') and a single unit. There is no data isolation for multi-tenancy.

**Evidence:**
- database.ts has a single company object hardcoded as 'Bigood Tecnologia LTDA'.
- No companyId field on any data type (Client, Subscription, Appointment, etc.).
- proxy.ts (middleware) checks session for access but does not scope data to companyId.
- The spec itself (AGENTS.md) mandates multitenancy but nothing implements it.

**Risk:** When the product serves multiple barbershops, every query and mutation must be scoped by companyId. The current code has zero awareness of this requirement. Retrofitting multitenancy will be expensive.

### M5. No API Rate Limiting, CSRF Protection, or Security Middleware

**What:** API routes have no request validation middleware, rate limiting, or CSRF protection.

**Evidence:**
- app/api/auth/login/route.ts — no rate limiting on login attempts (brute force possible).
- app/api/auth/signup/route.ts — no rate limiting on account creation.
- app/api/billing/activate/route.ts — no CSRF or idempotency key.
- proxy.ts is the only middleware, and it only handles route gating, not security.

**Risk:** The API routes are vulnerable to brute force attacks, CSRF attacks, and replay attacks. In production, this would be a critical security gap.

### M8. No Standardized Error Response Format

**What:** API routes return error messages in inconsistent shapes.

**Evidence:**
- login/route.ts: { message: '...' } with status 401
- signup/route.ts: { message: '...' } with status 400

### M9. No Loading/Error/Empty/Success State Coverage For Many Components

**What:** While some new components (e.g., client-auth-modal.tsx) handle all states, many older components lack complete state coverage.

**Evidence:**
- client-portal-shell.tsx has PortalLoading and PortalNotFound states for initial data load.
- But many admin views (agenda-view.tsx, financeiro-view.tsx, caixa-view.tsx) operate on the assumption that database data is always available.

### M10. No Explicit Client Portal Auth Guards on Sensitive Actions

**What:** While the client portal UI conditionally renders auth-required sections, there is no server-side enforcement for appointment creation or subscription — all logic executes client-side.

**Evidence:**
- createAppointment in client-portal-shell.tsx runs entirely in state — no fetch to server.
- subscribe in client-portal-shell.tsx saves to local state — no fetch to server.
- The handleBookingRequireAuth/handleCheckoutRequireAuth patterns gate the UI flow but there is no actual server call to validate or persist.

**Risk:** A malicious or technically savvy user could manipulate localStorage or client state to create appointments or subscribe without authentication. The app currently has no server-side enforcement because there is no server.

**What:** The project relies on TypeScript defaults without enabling stricter checks that catch common bugs.

**Evidence:**

## Recommended Action Plan

1. **Immediate (pre-backend):** Remove plain-text password storage. Hash client passwords even in mock mode. Rotate the .env secrets. Add a basic error boundary to the root layout.

2. **Short-term (backend planning):** Define API contracts (OpenAPI or spec docs) for all entities. Create the services/ layer with proper interfaces backed by mock implementations. Extract types from database.ts into types/.

3. **Medium-term (backend integration):** Implement real auth (HTTP-only cookies, JWT, bcrypt). Introduce companyId scoping. Add test infrastructure (vitest + testing-library). Split monolithic components.

4. **Ongoing:** Set up CI/CD pipeline (GitHub Actions + Vercel/Railway). Add Sentry for monitoring. Create a performance budget. Implement rate limiting and CSRF protection.
